# Tiqora API reference

Tiqora exposes four distinct API surfaces, all served by the same `tiqora-api`
process (default port `8000`) unless noted otherwise.

| Surface | Prefix | Audience | Auth |
|---|---|---|---|
| **Agent/admin API** | `/api/v1` | The Tiqora agent UI, and any first-party automation | Session cookie or `Authorization: Bearer <api-key>` |
| **Customer portal API** | `/api/portal` | The customer-facing self-service portal | Separate portal session cookie |
| **Compatibility layer** | `/znuny-compat` | Existing Znuny GenericInterface REST clients | Znuny-style `SessionID`, or `Authorization: Bearer <api-key>` |
| **MCP server** | separate process, default port `8001` | LLM agents / AI automation | `Authorization: Bearer <api-key>` (streamable-HTTP) |

See also: `/health`, `/ready`, `/metrics` (Prometheus) on the main API process
— these are unauthenticated and should be kept off any public-facing
listener (see [`../deploy/docker-compose.md`](../deploy/docker-compose.md)).

## Regenerating `openapi.json`

`docs/api/openapi.json` is a static snapshot of the FastAPI schema, generated by:

```sh
cd backend
uv run tiqora openapi -o ../docs/api/openapi.json
```

This does **not** require a live database, Redis, or Meilisearch connection —
`tiqora openapi` builds `create_app()` and calls `app.openapi()`, which only
inspects the route table (routes are registered at import time; the app's
`lifespan` — which does need those dependencies — never runs). Re-run this
after adding or changing any `/api/v1`, `/api/portal`, or `/znuny-compat`
route and commit the updated file.

The MCP server does not speak OpenAPI (it is not a REST API); its tool
surface is documented by hand in [`mcp.md`](mcp.md) from the tool registry in
`tiqora.mcp_server.server`.

## Authentication summary

- **Session cookie** (`/api/v1`, agent UI): `POST /api/v1/auth/login` sets an
  httpOnly cookie (name configurable via `TIQORA_SESSION_COOKIE`, default
  `tiqora_session`). Subsequent requests just need the cookie.
- **API key** (`/api/v1`, `/znuny-compat`, MCP): `Authorization: Bearer
  <key>`. Issued/revoked via the admin API
  (`/api/v1/admin/*` — API key management) or an existing session token can
  also be passed as a bearer token for CLI/automation convenience.
- **Portal session cookie** (`/api/portal`): separate cookie
  (`TIQORA_CUSTOMER_SESSION_COOKIE`, default `tiqora_customer_session`) from
  `POST /api/portal/auth/login`, scoped to a `customer_user`, not a `users`
  agent.
- **Znuny-style `SessionID`** (`/znuny-compat` only): `POST
  /znuny-compat/Session` (SessionCreate) returns a SessionID that is
  validated against Znuny's own `sessions` table when present, so a session
  created by Znuny itself still works against Tiqora's compat layer during
  parallel operation.
- **Auth method discovery**: `GET /api/v1/auth/methods` tells the login UI
  which of password / OIDC / SPNEGO / LDAP are enabled for this deployment
  (`TIQORA_OIDC_ENABLED`, `TIQORA_SPNEGO_ENABLED`, `TIQORA_LDAP_ENABLED`).

## Conventions

- **Pagination**: list endpoints that can return many rows (e.g. `GET
  /api/v1/tickets`) use `limit`/`offset` query parameters and return a
  `{"items": [...], "total": N}`-shaped body (see `PaginatedTickets` in
  `openapi.json`). CSV export variants (`*.csv` suffix) are unpaginated,
  streamed exports intended for reporting/download rather than UI paging.
- **Filtering**: resource list endpoints accept filter query parameters
  documented per-endpoint in the OpenAPI schema (queue, state, priority,
  customer, free-text, date ranges depending on the resource).
- **Errors**: standard FastAPI/Starlette shape,
  `{"detail": "<message>"}` for simple errors, or FastAPI's structured
  `{"detail": [{"loc": [...], "msg": ..., "type": ...}]}` for `422`
  validation errors. HTTP status codes follow normal REST conventions (`401`
  not authenticated, `403` authenticated but not permitted, `404` not found,
  `409` business-state conflict, `422` validation error).
- **IDs vs. ticket numbers**: `ticket_id` (`ticket.id`) is the internal
  numeric primary key used in URLs; `tn` (ticket number, e.g.
  `20260719000123`) is the human-readable identifier shown in the UI and in
  email subjects. Most endpoints take `ticket_id`; ticket-number lookup goes
  through search.
